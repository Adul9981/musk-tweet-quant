# 工具更新日志

> 记录每次工具升级的原因、内容、验证结果和遗留问题。
> 目的：让每一次改动都有据可查，积累工具演进的完整脉络。
> 格式：倒序（最新在前）

---

## 设计说明：这个日志记录什么

每条记录包含以下维度，缺一不可：

| 字段 | 说明 | 为什么重要 |
|------|------|----------|
| **触发原因** | 什么事情促使这次更新 | 避免「为改而改」，每次改动都要有根据 |
| **问题描述** | 旧工具哪里不够用 | 明确痛点，才知道改了有没有解决问题 |
| **变更内容** | 具体改了哪些文件/函数/规则 | 可追溯，方便回滚或继续扩展 |
| **验证方式** | 怎么确认改动有效 | 防止「感觉改好了」但实际没用 |
| **实测效果** | 真实运行结果或数据 | 最直接的证明 |
| **遗留问题** | 这次没解决的、下次要改的 | 不让问题被遗忘，建立持续改进队列 |
| **关联规则** | 对应 RULES.md 的哪条规则 | 工具和规则保持同步 |

## v2.4 — 2026-05-27

### 触发原因
用户提出「电报机器人的预警信息能优化吗？」，发现原 `alert_bot.py` 存在根本性设计缺陷：每条推文都推一次通知，信息含量为零，用户会立刻关掉。

### 问题描述
旧 `alert_bot.py` 的三个根本问题：
1. **噪音极高**：马斯克每天发 40+ 条推文，每条都通知 → 用户关闭通知
2. **无统计维度**：只推「马斯克发推了！+ 推文内容」，没有 µ、价值比、操作建议
3. **无决策导向**：纯信息推送，不回答「我现在应该做什么」

### 变更内容

**新建 `scripts/musk_tgbot.py`**（完整重写，替代 `alert_bot.py`）：

三套预警系统协同：

| 系统 | 触发方式 | 内容 |
|------|---------|------|
| 时间窗口播报 × 6 | cron 定时 | 与会话节奏对齐的结构化分析 |
| 到期倒计时 × 4 | check 内自动 | T-72h/24h/12h/6h，各发一次 |
| 异常事件触发 × 4 | check 内检测 | µ漂移/价格异常/节奏异常/连续降温 |

**6个时间窗口播报**：
- `prebuild`（BJ 11:30）：深夜会话前预告，最强建仓窗口，含价值比全表+三层入场建议
- `peak_check`（BJ 12:30）：深夜会话内检查，90min实际发推 vs 历史预期，检测偏热/沉默
- `peak_end`（BJ 16:30）：深夜会话结束，µ更新，提示死区即将到来
- `deadzone`（BJ 17:30）：死区必检三项（连续降温/仓位健康/保护仓），含铁律提醒
- `morning_pre`（BJ 20:00）：晨间会话预告，保存快照
- `morning`（BJ 21:00）：晨间建仓完整分析，含剩余时间三档建议

**4个到期倒计时**：
- T-72h：首次预测窗口，明确标注 t-3 精度约52%，建议只建保护仓
- T-24h：精确预测窗口，明确标注 t-1 精度约85%，完整入场决策
- T-12h：止盈评估，>75¢ 建议卖出50%锁利
- T-6h：最后理性操作窗口，亏损仓位铁律提醒

**4个异常触发**：
- 会话内偏热（90min发推 >180% 预期）→ 不追仓 + 止盈提示
- 会话内沉默（90min发推 <20% 预期）→ µ可能虚高警告
- µ漂移（|Δµ| ≥ 15条，约0.75档）→ 调仓参考提示
- 价格异常（中心区间>35¢且VR<1.0）→ 自动列出价值比更优的相邻区间
- 连续降温（近2天均 <均值30%）→ 必须今日死区重算µ

**新建 `/musk-tgbot.md`**（Telegram 机器人规则库）：
- 完整设计说明（三套系统 + 触发逻辑 + 冷却规则）
- 与 RULES.md、RETROSPECTIVE.md 的联动索引
- Cron 配置文档
- 维护和进化指南

**状态文件 `scripts/tgbot_state.json`**：
- 追踪 last_mu（µ漂移检测）、session_snapshots（会话内异常检测）
- period_id（到期预警每期只发一次）、alerts_sent（冷却机制）

### 验证方式
```bash
# 测试连接
python3 musk_tgbot.py test

# 测试窗口播报
python3 musk_tgbot.py window prebuild

# 测试异常检测（含到期倒计时）
python3 musk_tgbot.py check
```

### 遗留问题
- [ ] 上午会话（BJ 20:30）暂未做会话内节奏检查（只有深夜会话有），后续可补
- [ ] T-48h 预警节点暂缺（µ精度提升的中间观察点）
- [ ] 止盈/止损建议目前无法知道用户的实际入场价，提示只能是价格参考而非个性化计算
- [ ] `tgbot_state.json` 的 session_snapshots 依赖 prebuild/morning_pre 触发时保存，若那两次 cron 失败则会话内检查无数据

### 关联规则
- RULES.md § 2.2「µ不确定性随时间增大」（到期倒计时精度标注）
- RULES.md § 2.3「会话缺席的µ影响」（深夜会话偏冷告警的µ下移估算）
- RULES.md § 4.3「🔴 连续降温触发规则」（连续降温预警）
- RULES.md § 5.2「死区铁律」（死区必检三项的核心内容）
- RULES.md § 1.1.2「中心区间为什么高估」（价格异常告警）

## v2.2 — 2026-05-27

### 触发原因
用户提出：「当前180-199区间已涨至44¢，盈亏比不如两侧区间，我们在决策时是否应考虑入场时机和各区间盈亏比？」

回测结论同步触发：33期数据显示模型存在+0.3档系统性偏高，t-1时±1档覆盖率85%（近期），说明覆盖2个档位比押注单一区间更合理。

### 问题描述
旧工具的三个不足：
1. `cmd_scan()` 只显示「高价值Top5」，不判断中心区间是否被高估，用户需要自己计算
2. `cmd_scan()` 没有推荐具体仓位结构，用户看完数据仍不知道主仓/保护仓分别放哪里
3. `cmd_check()` 只分析持仓区间本身，不横向对比相邻区间，用户不知道有没有更好的替代选择

### 变更内容

**`scripts/musk_engine.py`**

新增函数：
- `vr_label(vr)` — 统一价值比标签（⭐高赔率低估 / ✅低估 / 🟡合理 / 🟠略高估 / ❌高估）
- `suggest_entry(analysis, mu_a)` — 自动生成三层仓位结构：
  - 主仓：model_pct>10% 中价值比最高的区间
  - 保护仓：中心落点下方1档（基于+0.3档系统偏高的回测结论）
  - 高赔率仓：价格≤5¢ 且 VR≥2.0（可选）
  - 中心区间>35¢ 且 VR<1.0 时触发「定价偏高」警告

`cmd_scan()` 改动：
- 「高价值Top5」→「各区间价值比全表」（显示所有 model_pct>2% 的区间）
- 新增「📋 入场结构建议」输出模块
- 中心区间高估时自动提示「⚠️ 建议看两侧区间」

`cmd_check()` 改动：
- 新增「🔄 相邻区间对比」表格（对比持仓区间前后±2档）
- 自动标注「保护仓候选」（下方1档）
- 有更高VR相邻区间时，自动输出换仓提示

**指令文件**
- `/musk-scan.md` — 更新输出解读，增加中心区间高估处理
- `/musk-plan.md` — 仓位结构从「主仓80-90%」→「三层结构（主仓50-70% + 保护仓20-30% + 高赔率≤5%）」
- `/musk-check.md` — 增加相邻区间对比和换仓判断框架

### 验证方式
实际运行 `scan` 和 `check "180-199" 0.44` 并检查输出。

### 实测效果
```
scan 输出（May 22-29 市场，2026-05-27）：
  中心落点：160-179  市价 6.9¢  VR 9.17x  ⭐ 高赔率低估
  入场结构建议：
    主仓 → 160-179  VR 9.17x
    保护仓 → 无（中心已在最低档）
    总体评估：✅ 有入场价值

check "180-199" 0.44 输出：
  浮亏 -62.6%  |  µ=173 偏出区间
  相邻区间：160-179  VR 9.17x >> 持仓 2.05x
  💡 提示换仓至 160-179
```

### 遗留问题
- [ ] `suggest_entry()` 保护仓逻辑：当中心落点已在最低档时，「无保护仓」的情况应有更明确的替代建议（如「上方1档」也可作为两侧覆盖）
- [ ] 高赔率仓候选目前只排除主仓和保护仓，但没有排除与主仓重叠的区间
- [ ] `cmd_check()` 多仓位批量输入时，相邻对比表会重复显示，需要优化格式
- [ ] 换仓成本估算：目前只提示「需权衡换仓成本」，未量化换仓的价差损耗

### 关联规则
- RULES.md § 1.1.1「预测正确 ≠ 下注正确」
- RULES.md § 1.1.2「中心区间为什么经常被高估」
- RULES.md § 3.2「入场条件检查清单（第四步：价值比横向比较）」
- RULES.md § 3.3「各区间价值比评估方法」
- RULES.md § 4.1「仓位集中原则（含两档覆盖策略）」
- RULES.md § 10.2「中心区间定价过高时，转买两侧」
